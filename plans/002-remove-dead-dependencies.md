## Why this matters

Eight dependencies in `package.json` are not imported anywhere in `src/`. They
add install time, supply-chain/audit surface, and bundle weight for zero
benefit, and they create false signals (e.g. an unused `@prisma/adapter-pg@^6`
that looks like a version mismatch against `@prisma/client@^5`, and an unused
pre-release `mux@^0.0.2`). Removing confirmed-dead dependencies is a clean,
low-risk win with a build-passes verification story.

## Current state

Each package below was verified unused at commit `5f3425a` with
`grep -rln "<name>" src` returning **zero** import sites. Confirmed dead:

Relevant repo rule (from `CLAUDE.md`): **if `package-lock.json` changes, the
`npmDepsHash` in `flake.nix` must be regenerated**, or preview/CI deployments
fail. This plan removes packages, so the lockfile *will* change.

## Steps

### Step 1: Re-confirm each package is unused

For every package in the table, run `grep -rln "<pkg>" src`. Each must return
**no output**. Build a list of only those that are confirmed unused right now.

```sh
for p in @prisma/adapter-pg @mux/mux-player-react mux timeago.js querystring yamljs three @types/three @types/aws-sdk; do
  echo "== $p =="; grep -rln "$p" src || echo "  (unused)"
done
```

**Verify**: every package prints `(unused)`. If any package now has an import
site, REMOVE IT FROM THIS PLAN'S REMOVAL LIST and note it in your report (do not
delete a package that is in use).

> Note on `mux` vs `@mux/*`: the grep for `mux` will also match `@mux/...`
> import lines. Confirm specifically that there is no bare
> `from 'mux'` / `require('mux')` before removing the `mux` package:
> `grep -rn "from 'mux'\|require('mux')\|from \"mux\"" src` must be empty.

### Step 2: Remove the entries from `package.json`

Delete the corresponding lines from `dependencies` / `devDependencies`. Do not
touch any other line. Keep JSON valid (no trailing commas).

**Verify**: `node -e "JSON.parse(require('fs').readFileSync('package.json','utf8')); console.log('valid')"` → prints `valid`.

### Step 3: Regenerate the lockfile

Run `npm install`. This rewrites `package-lock.json` to drop the removed
packages and their now-orphaned transitive deps.

**Verify**: `npm install` exits 0. `git diff --stat package-lock.json` shows the
lockfile changed.

### Step 4: Typecheck, build, and test

Run, in order:
- `npx tsc --noEmit` → exit 0
- `npm run build` → exit 0
- `npm test` → all pass

If any fails, a "dead" package was actually referenced indirectly (e.g. via a
config file outside `src/`). Identify it, restore that one package, and report.

**Verify**: all three commands succeed.

### Step 5: Update the Nix deps hash

Because `package-lock.json` changed, regenerate the hash and write it into
`flake.nix`:

```sh
nix run nixpkgs#prefetch-npm-deps package-lock.json
```

Copy the printed `sha256-...` value into the `npmDepsHash` field in `flake.nix`
(replace the existing value only).

> If the `nix` command is unavailable in your environment, STOP and report that
> `flake.nix`'s `npmDepsHash` still needs updating — do not guess a hash. A wrong
> or stale hash breaks preview/CI deployments.

**Verify**: `grep -n "npmDepsHash" flake.nix` shows the new hash; `git diff flake.nix` shows only that line changed.
